# 项目总体规范

## 1. 文档目的

本规范用于明确 `astro-minblog` 当前仓库的系统定位、包划分、职责边界、核心能力与开发约束，为后续基于 AI 的规范驱动开发提供统一的基线。

本规范覆盖以下问题：

- 当前项目由哪些包和应用组成。
- 各包之间如何协作。
- 当前系统的核心目标与边界是什么。
- 后续需求在进入设计或实现前，应如何映射到仓库结构。

## 2. 仓库形态

当前仓库是一个 PNPM monorepo，工作区由 `pnpm-workspace.yaml` 定义，包含：

- `packages/*`
- `apps/*`

仓库顶层 `package.json` 的脚本表明，该仓库以 monorepo 形式统一管理开发、构建、格式化、lint、预览与 postinstall 构建流程。

## 3. 系统定位

根据顶层描述与各包清单，当前项目定位为：

一个以 Astro 为基础的模块化博客主题与示例应用 monorepo，提供以下能力：

- 核心博客主题与页面注入能力。
- AI 驱动的内容理解、RAG 检索与对话能力。
- 多渠道通知能力。
- 面向博客站点的 CLI 与 AI 数据处理工具链。
- 一个消费全部能力的示例博客应用 `apps/blog`。

## 4. 包划分总览

当前仓库的主能力基线可概括为 4 个核心 package、1 个辅助共享契约 package，与 1 个 app：

### 4.1 `packages/core`

核心主题包，负责：

- Astro integration 入口。
- 路由注入。
- 布局、组件、页面、样式、插件与通用工具。
- 通过 virtual module 将站点配置、社交链接、好友链接、样式与可选 AI 组件暴露给运行时。

### 4.2 `packages/ai`

AI 能力包，负责：

- 聊天请求处理。
- RAG 检索、证据整合、提示词组装与流式响应。
- Provider 管理、缓存、中间件、扩展、结构化输出与聊天组件。
- Dev server 命令 `astro-ai-dev`。

### 4.3 `packages/notify`

通知能力包，负责：

- 评论与 AI 对话相关通知。
- Telegram / Webhook / Email(Resend) 多渠道分发。
- 通知模板、provider 抽象与 comment webhook 处理。

### 4.4 `packages/cli`

命令行工具包，负责：

- `astro-minimax` CLI 入口。
- 项目初始化。
- 文章创建、列表、统计。
- AI 数据处理与事实、扩展、作者画像相关命令。
- 数据清理与 Git hooks 管理。

### 4.5 `packages/knowledge-model`（辅助共享契约包）

该包用于为 `ai` 与 `cli` 提供共享知识数据契约。它是当前仓库中的辅助支撑包，而不是与 `core / ai / notify / cli` 同层级的最终功能入口包。其职责包括：

- 定义知识语料、摘要、向量、bundle 等运行时数据结构。
- 为 CLI 与 AI 包提供共享 schema 与类型契约。

### 4.6 `apps/blog`

示例博客应用，负责：

- 提供站点级配置、内容数据与部署配置。
- 通过 `@astro-minimax/core` 集成主题与路由。
- 通过 Cloudflare Pages Functions 暴露 AI 与通知 API。
- 作为 monorepo 中各 package 的集成消费端。

## 5. 系统边界

### 5.1 当前系统明确包含的能力

- 博客站点渲染与页面路由。
- 基于 Markdown / MDX 的内容管理。
- 多语言博客内容目录。
- 标签、分类、系列、归档、友情链接、项目页等信息架构。
- AI 聊天与知识检索。
- 站点内容 AI 数据加工。
- 评论与 AI 对话通知。
- 面向博客站点的 CLI 工作流。
- Cloudflare Pages 部署适配。

### 5.2 当前系统不应默认假定的能力

以下能力在没有新增实现前，不应在规范或需求中视为已支持：

- 多 app 的独立运行时编排。
- 通用 CMS 后台管理。
- 数据库持久化内容编辑。
- 复杂工作流引擎或任务调度中心。
- 任意框架消费端兼容性保证。

## 6. 关键架构原则

### 6.1 Core 负责主题与站点骨架，App 提供内容与配置

`apps/blog` 不以自定义 `src/pages/` 为主来承载业务页面，核心页面由 `packages/core/src/integration.ts` 通过 `injectRoute()` 注入。应用侧主要负责：

- `src/config.ts`：站点配置。
- `src/constants.ts`：社交与分享链接。
- `src/data/`：内容与好友链接。
- `functions/api/`：Cloudflare Pages API 封装。

### 6.2 AI、通知、CLI 均为可独立演进的能力包

当前架构将 AI、通知、CLI 从核心主题中分离，意味着后续需求应优先判断其归属：

- 页面展示与主题表现 → `core`
- 对话、检索、知识数据、提示词、provider → `ai`
- 消息发送与 webhook → `notify`
- 本地工具链与批处理 → `cli`

### 6.3 共享知识数据结构由 `knowledge-model` 统一定义

知识 bundle、corpus、passages、summaries、vector 索引等结构，不应在 `ai` 或 `cli` 内各自重复定义。这里的重点是：`knowledge-model` 负责共享契约，不负责作为站点主能力入口直接被最终 app 使用。

## 7. 当前总体目标

当前项目的总体目标可归纳为：

1. 提供一个可发布、可部署、可扩展的 Astro 博客主题体系。
2. 提供完整的 AI 内容理解与对话能力，并与博客内容深度耦合。
3. 提供面向真实站点运营的通知与工具链能力。
4. 通过示例 app 演示 package 的集成方式与最佳实践。
5. 为未来持续迭代提供明确模块边界，降低修改耦合度。

## 8. 当前总体非目标

以下目标不属于当前基线，除非后续需求明确提出并落地实现：

- 将仓库改造成通用 SaaS 平台。
- 为所有 package 建立独立 UI 控制台。
- 将 `apps/blog` 改造成“唯一真相源”，反向削弱 packages 的模块化职责。
- 将 AI 能力写死到 core，破坏 package 分层。

## 9. SDD 使用约束

后续在本仓库中进行 SDD 开发时，需求必须先明确以下信息：

- 需求归属哪个包或 app。
- 是新增能力、增强现有能力还是修复问题。
- 影响哪些目录层级。
- 是否涉及运行时链路变更。
- 验收是看页面行为、API 行为、CLI 输出还是构建产物。

任何需求如果无法回答上述问题，说明规范尚不完整，不应直接进入实现。

## 10. 规范维护要求

- 新增 package 时，必须更新本文件与《包与目录结构规范》。
- 调整核心链路时，必须更新《核心链路与功能特性规范》。
- 调整目标、约束或验收口径时，必须更新《需求、目标与验收规范》。
- 若 `apps/blog` 不再是唯一示例应用，应补充 app 层规范并重写本文的系统边界部分。
