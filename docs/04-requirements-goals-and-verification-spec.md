# 需求、目标与验收规范

## 1. 文档目标

本规范用于把当前仓库的需求表达、目标拆解、验收口径与检验要求固定下来，作为后续 SDD 工作的执行基线。

它回答的问题是：

- 当前项目各能力域的目标是什么。
- 后续需求应该如何描述才足够可执行。
- 验收时必须检查什么，什么不算完成。

## 2. 需求表达规范

后续提交到本仓库的需求，至少必须覆盖以下维度：

### 2.1 需求归属

必须明确需求归属：

- `core`
- `ai`
- `notify`
- `cli`
- `knowledge-model`
- `apps/blog`

如果一个需求横跨多个包，必须说明主归属包和受影响协作包。

### 2.2 需求类型

必须明确属于以下哪一类：

- 新增能力
- 现有能力增强
- 缺陷修复
- 结构重组
- 规范补全

### 2.3 影响面

必须说明影响面属于以下哪一种：

- 页面渲染
- 配置结构
- API 行为
- CLI 行为
- 数据结构
- 构建链路
- 部署行为
- 文档规范

### 2.4 验收对象

必须说明最终看什么判断完成：

- 页面行为
- API 返回
- CLI 输出
- 生成数据文件
- 构建结果
- 文档内容

没有明确验收对象的需求，不应直接进入实现阶段。

## 3. 项目级目标规范

## 3.1 `core` 目标

- 提供稳定的主题 integration 能力。
- 提供统一的页面注入机制。
- 提供稳定的 virtual module 装配机制。
- 提供可复用的布局、页面、组件、样式与内容处理插件。

### `core` 验收重点

- 路由是否按 feature 开关正确注入。
- 类型定义是否与 virtual modules 保持一致。
- 页面和样式变更是否仍符合 app 消费模式。

## 3.2 `ai` 目标

- 基于博客内容提供可运行的 AI Chat 与 RAG 能力。
- 提供缓存、限流、provider failover、扩展与通知集成能力。
- 保持 server handler、search、intelligence、provider 管理之间的清晰分层。

### `ai` 验收重点

- `/api/chat` 能否完成请求处理。
- 检索、证据分析、流式输出是否仍可闭环。
- provider fallback 与 timeout 逻辑是否仍然成立。
- 运行时知识 bundle 是否能被正确加载。

## 3.3 `notify` 目标

- 为评论与 AI 对话事件提供可靠的多渠道通知。
- 让渠道、模板和核心分发逻辑保持可替换、可扩展。
- 保证通知失败不拖垮主流程。

### `notify` 验收重点

- webhook 入口是否能正常调用包内处理器。
- 多 provider 情况下是否能返回汇总结果。
- 无 provider、单 provider、部分 provider 失败时是否都能稳定返回。

## 3.4 `cli` 目标

- 提供可直接服务于博客开发与内容运营的 CLI。
- 对工作目录、内容目录和数据目录做明确约束。
- 让命令帮助信息、行为和输出保持一致。

### `cli` 验收重点

- 命令是否已在入口注册。
- 帮助信息是否同步更新。
- 子命令是否对 blog 根目录前提做校验。
- 实际命令输出是否与预期匹配。

## 3.5 `knowledge-model` 目标

- 作为知识 bundle 的唯一共享契约源。
- 为 `ai` 与 `cli` 提供稳定类型边界。

### `knowledge-model` 验收重点

- schema 与类型是否足够表达运行时知识数据。
- 上层包是否复用了这里的结构，而不是复制漂移。

## 3.6 `apps/blog` 目标

- 作为完整示例站点，消费所有关键 package 能力。
- 提供站点配置、内容数据、API 封装与部署适配。
- 为 monorepo 提供真实运行验证场景。

### `apps/blog` 验收重点

- 配置是否能被 integration 正确消费。
- 内容目录与 schema 是否匹配。
- API 包装层是否正确连接底层 package。
- `build`、`preview`、`typecheck` 等脚本是否符合约定。

## 4. 功能域需求规范

## 4.1 页面与主题需求

当需求涉及页面、布局、样式、导航、内容展示时，必须明确：

- 是否属于主题共性能力。
- 是否由 route injection 生效。
- 是否受 `SITE.features` 或 preferences 影响。
- 是否需要同时修改样式、组件、页面与类型。

### 验收要求

- 页面可被正确访问。
- feature 开关与显示行为一致。
- 如涉及 UI 变化，应至少进行手动页面验证。

## 4.2 AI 需求

当需求涉及 AI Chat、知识检索、事实、扩展、提示词、provider 时，必须明确：

- 影响 API 入口还是内部 pipeline。
- 是否影响知识 bundle 结构。
- 是否影响 timeout、cache、health、fallback。
- 是否会改变响应结构、状态流或通知行为。

### 验收要求

- API 能正常调用。
- 错误路径和 fallback 路径仍成立。
- 若变更了数据结构，相关 bundle 或运行时数据必须同步验证。

## 4.3 通知需求

当需求涉及通知事件、模板、渠道时，必须明确：

- 是新增事件、增强模板还是接入新 provider。
- 是否影响 comment 与 ai-chat 两类事件。
- 是否可能引入阻塞式错误。

### 验收要求

- 事件能正确路由到模板与 provider。
- 部分失败时仍能生成可理解的结果。
- 主流程不被通知异常中断。

## 4.4 CLI 需求

当需求涉及 CLI 命令时，必须明确：

- 命令属于哪个子树。
- 输入参数与输出格式是什么。
- 运行前提目录是什么。
- 失败时的错误提示是什么。

### 验收要求

- 命令可实际执行。
- 帮助信息、参数解析、错误输出均正确。
- 若生成文件，需验证文件实际落地。

## 4.5 文档规范需求

当需求涉及 docs 或规范本身时，必须明确：

- 文档服务对象是谁：实现者、评审者、维护者还是 AI。
- 文档是总览、结构说明、链路说明还是验收说明。
- 文档依据的是当前代码还是未来规划。

### 验收要求

- 文档路径正确。
- 标题与层级结构清晰。
- 内容与当前代码基线一致。
- 不将未实现能力写成已完成事实。

## 5. 检验要求总表

## 5.1 文档类改动

至少执行：

- 文件存在性检查。
- 文档内容抽样检查。
- 若仓库对 markdown 有静态检查，可补充执行；若无，则至少保证新增文件结构清晰、可读取。

## 5.2 代码类改动

至少执行：

- 相关文件 diagnostics 检查。
- 相关测试或最接近的执行验证。
- 与改动类型对应的手动验证。

## 5.3 API 类改动

至少执行：

- 请求真实调用或最接近的模拟调用。
- 响应结构检查。
- 错误路径检查。

## 5.4 CLI 类改动

至少执行：

- 真实运行命令。
- 检查终端输出。
- 若命令生成文件，检查文件落地结果。

## 5.5 构建与集成类改动

至少执行：

- `typecheck` 或等价检查。
- 必要时执行 `build`。
- 确认集成端未因接口漂移而失效。

## 6. 完成定义

在本仓库中，一个需求只有在以下条件全部满足时才能被视为完成：

1. 需求描述的目标已被实现或文档化完成。
2. 影响范围内的关键路径已经验证。
3. 没有引入新的明显结构冲突或契约漂移。
4. 验收对象已经被实际检查，而不是主观推断。
5. 若属于规范文档需求，则文档已落到 `docs/` 且结构完整可读。

## 7. 本次文档任务的专用验收口径

针对当前“在 `docs/` 下生成多个详细且全面的规范文档”这一任务，完成标准为：

- `docs/` 目录存在。
- 至少包含项目总览、包/目录结构、核心链路/特性、需求/验收四类文档。
- 文档内容覆盖包划分、目录结构、文件作用、核心链路、功能特性细节、需求与目标、检验要求。
- 文档内容基于当前仓库真实结构，而不是抽象模板。

## 8. 维护要求

- 后续任何重大结构变更都必须回写到本规范。
- 若未来引入更多 app 或 package，需要补充新的目标、需求与验收条目。
- 若 CI、测试策略、部署方式发生变化，应同步更新检验要求章节。
